# Module 3: Your First Capture

vault-intake takes text as input. It doesn't fetch URLs or download videos -- you get the content first, then pipe it in. This is the right mental model: vault-intake is a classifier, router, and formatter, not a scraper.

## Step 1: Get your content as text

Pick something you actually want to keep. Good first sources:

- A YouTube transcript (use yt-dlp: `yt-dlp --write-auto-subs --skip-download "URL"`, then feed the `.vtt` or `.srt` file)
- An article or blog post (copy the full text, paste into a `.txt` file)
- Notes from a meeting or conversation (paste your rough notes into a file)
- A long thread or document you want in your vault

Save the content as a plain text file. vault-intake reads from stdin or a file path.

<Insight>
The best first capture is something where you already know the content well. When vault-intake produces a note on content you know, you can immediately tell whether the classification is accurate, whether the domain assignment makes sense, and whether the refined output captured the key points. Generic "test" content gives you no calibration signal.
</Insight>

## Step 2: Run the capture

**Via Claude Code skill (primary path):**

Open a Claude Code session and type `/vault-intake`. The skill will load and prompt you for input. Paste your content directly into the chat, or tell the skill the file path.

**Via standalone CLI:**

```bash
# From a file:
uv run scripts/intake.py --vault /path/to/vault --input /path/to/content.txt

# From stdin:
cat content.txt | uv run scripts/intake.py --vault /path/to/vault

# Dry run (see the output without writing to vault):
uv run scripts/intake.py --vault /path/to/vault --input content.txt --dry-run

# Skip confirmation prompt:
uv run scripts/intake.py --vault /path/to/vault --input content.txt --yes
```

<Action>
Run the dry run first (`--dry-run`). It shows you exactly what vault-intake would write without touching your vault. Confirm the classification and output look right before committing the write.
</Action>

## Step 3: Read the output

vault-intake writes a Markdown file to your vault. The structure:

```markdown
---
schema_version: '1.0'
source_type: paste
source_uri: ''
captured_at: '2026-05-02'
processed_by: /vault-intake
confidence: 0.82
title: your-note-title
date: '2026-05-02'
type: note
domain: ai-tools
tags:
  - ai-tools
  - knowledge-management
---

{refined or original body text}

## Possible next steps
- Review the key points [Signal: action item detected]

## Original capture
{verbatim original input, only appears when refinement changed the text}
```

The file lands in a folder inside your vault determined by the `(type, routing_mode)` combination -- for example, `resources/` for reference material or `insights/` for analytical notes when using PARA routing.

<KeyData>
The `confidence` frontmatter field tells you how certain vault-intake was about the domain classification. A value below 0.6 (the default threshold) means it wasn't confident enough to classify -- it will flag this for your review rather than guessing. If you see low-confidence results on content that clearly belongs to one of your domains, your domain descriptions may need to be more specific.
</KeyData>

## What vault-intake does NOT add automatically

Some things that are NOT in the output:
- **Summary sections** -- the refined body IS the summary; there's no separate `## Summary` heading
- **Wikilinks to existing notes** -- the dry-run surfaces suggested wikilinks for your review; they are not auto-inserted into the note
- **A `## My Notes` section** -- the `## Original capture` section appears only when refinement changed the input; there's no empty annotation placeholder

If your output looks different from this -- no frontmatter, no domain assignment, an error message -- Module 4 is where you report that.

<WatchOut>
If vault-intake can't determine the content type or domain, it will ask for clarification rather than guess. This is expected behavior for ambiguous content. Your first capture should be something clearly within one of your configured domains.
</WatchOut>
