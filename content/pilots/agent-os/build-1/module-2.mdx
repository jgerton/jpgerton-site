# Module 2: Configure Your Vault

vault-intake reads a `## Vault Config` section from your vault's `CLAUDE.md` to understand how to classify and route your captures. This is not your global `~/.claude/CLAUDE.md` -- it's the `CLAUDE.md` file inside your Obsidian vault directory.

If you don't have a `CLAUDE.md` in your vault yet, create one. vault-intake looks for a heading called `## Vault Config` followed by a YAML block.

## The config block

Add this to your vault's `CLAUDE.md`:

````markdown
## Vault Config

```yaml
vault_path: /absolute/path/to/your/vault
classification_mode: fixed_domains
routing_mode: para
domains:
  - slug: your-first-domain
    description: One sentence describing what this domain covers
  - slug: your-second-domain
    description: One sentence describing what this domain covers
```
````

Three fields are required: `vault_path`, `classification_mode`, and `routing_mode`. If `classification_mode` is `fixed_domains`, `domains` is also required.

<WatchOut>
`vault_path` must be an absolute path, not a relative one. `~/Documents/Obsidian/MyVault` works on Mac/Linux. On Windows use forward slashes: `C:/Users/yourname/Documents/MyVault`.
</WatchOut>

## Filling in your domains

Each domain needs two sub-fields:

- **slug** -- a short kebab-case identifier used in filenames and tags (e.g., `ai-tools`, `knowledge-management`)
- **description** -- one sentence describing what this domain covers, written in plain language

vault-intake uses the description to classify incoming content. A specific description produces more accurate classification than a broad one.

<Insight>
Write domain descriptions as if you're explaining to a colleague what goes in that folder. "Notes on building Skool communities, freemium conversion, and member retention" is more useful than "community." The description is read by the model during classification -- specificity matters.
</Insight>

**Example for an indie hacker / builder:**

```yaml
domains:
  - slug: ai-tools
    description: Tools, workflows, and techniques for using AI in daily knowledge work and content creation
  - slug: community-building
    description: Building and growing online communities, Skool mechanics, member engagement, and freemium conversion
  - slug: product-development
    description: Building software products as an indie hacker, including technical and business decisions
  - slug: knowledge-management
    description: Obsidian, note-taking systems, PKM methods, and how to structure a personal knowledge base
```

Your domains should reflect what is actually in your vault. Start with 3-5 domains. You can add more later.

## Mode choices

**`classification_mode`:** Use `fixed_domains` if you have a defined set of topics (recommended for most users). Use `emergent` if you want vault-intake to infer domain categories from content without a predefined list.

**`routing_mode`:** Use `para` to route notes into a PARA-style folder structure (Projects, Areas, Resources, Archives). Use `emergent` to let vault-intake decide the folder based on content type.

The most common pairing is `fixed_domains` + `para`. If you're not sure, use that.

## Optional fields

```yaml
language: en                              # default: en
skip_notebooklm: false                    # default: false; set true to skip NotebookLM integration
refinement_enabled: true                  # default: true; set false to skip AI refinement step
classification_confidence_threshold: 0.6  # default: 0.6; lower = more permissive classification
notebook_map:                             # optional; map domain slugs to NotebookLM notebook IDs
  ai-tools: your-notebook-id-here
```

For the alpha pilot, you only need the three required fields plus your domains. Leave the optional fields at defaults unless you have a specific reason to change them.

## A complete working example

```yaml
vault_path: /Users/jon/Documents/Obsidian/Vault
classification_mode: fixed_domains
routing_mode: para
domains:
  - slug: ai-tools
    description: Tools, workflows, and techniques for using AI in daily knowledge work and content creation
  - slug: community-building
    description: Building and growing online communities, Skool mechanics, member engagement, and freemium conversion
  - slug: product-development
    description: Building software products as an indie hacker, including technical and business decisions
  - slug: knowledge-management
    description: Obsidian, note-taking systems, PKM methods, and how to structure a personal knowledge base
```

## Validate your config

After writing your config, validate it before running a capture:

```bash
uv run scripts/resolve_config.py /path/to/your/vault/CLAUDE.md
```

This prints the resolved config as JSON. If it errors, the message will tell you which required field is missing or malformed.
