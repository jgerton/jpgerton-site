# Module 4: Friction Reporting

Friction is any moment where the tool slowed you down, confused you, required a workaround, or made you want to close the terminal. You've now run at least one real capture. This module is where you report what felt off.

## What counts as friction

**Slow:** The capture took longer than you expected. The output required more editing than felt worth it.

**Confusing:** You weren't sure what to do next. The documentation (or lack of it) left you guessing. The output format was unexpected.

**Broken:** Something didn't work. An error message appeared. The output was wrong or empty.

**Annoying:** Something worked but felt unnecessarily awkward. A required step that should be optional. A default that doesn't match how anyone would actually use the tool.

<Insight>
Friction is more valuable than "it worked great" feedback. A tool that works but has friction will be abandoned. A tool with friction that's reported can be fixed. If your experience was mostly smooth, that's also worth reporting -- but think hard before concluding there was no friction.
</Insight>

## Three categories of friction

Not all friction is the same kind. Before reporting an item, consider which bucket it's in:

**Config issue (you can fix it):** Your Vault Config block is misconfigured, or a setting doesn't match your vault setup. Solution: adjust the config and re-run. Still worth reporting so the defaults can be improved.

**Workflow habit (fixable with practice):** The tool works correctly but your mental model of how to use it needs adjustment. Example: expecting vault-intake to know which folder a capture belongs in before you've configured it.

**Product issue (needs a fix in vault-intake):** The tool behaved in a way that can't be fixed by configuration. Wrong output format, missing feature, misleading error message, unclear documentation.

You don't need to categorize every item in your response. If you're not sure, just describe what happened -- the categorization is useful context, not a requirement.

## How to write a good friction report

Specific and reproducible beats general and impressionistic.

**Less useful:** "The output wasn't great."

**More useful:** "When I captured a YouTube video longer than 60 minutes, the summary skipped the second half of the content entirely. Shorter videos produced complete summaries."

Each friction item should be one clear statement. List them one per line -- don't try to group or organize them.

List your friction items, one per line. Include anything that slowed you down, confused you, or made you work around the tool rather than with it.

There's no minimum or maximum. Two specific items are more useful than ten vague ones.
